<objective>
Re-home the still-live fulfillment functionality into the reachable v4.0 Order step (SC1–SC4,
ORDER-04/05). Today the canvas-PNG downloads and the Diamond Drills USA drill cart are wired
ONLY into the dormant `Step3Canvas` (panel-3 already renders `SuppliesScreen` because
`USE_NEW_SUPPLIES` is true), so the customer cannot actually reach them. This plan moves their
CALL SITE onto `OrderScreen` props — App stays the state owner (the handlers keep their App
home, D-02) — and restructures OrderScreen into the two honest task sections of D-07.

This is the ADDITIVE re-home step of the strangler: nothing is deleted here (Step3Canvas,
flags.ts, and the coupled modals are removed in 26-03, which depends on this plan). The app
ships green at this commit — OrderScreen gains the fulfillment actions while the dormant
Step3Canvas branch still compiles behind the untouched flag.

Purpose: make ORDER-04 (canvas PNGs alongside the JSON packet) and ORDER-05 (the single
Diamond Drills USA cart) real, reachable features on the Order step, honestly framed as a
two-vendor self-serve handoff.

Output: OrderScreen two-section UI + per-task state; App wiring (new `handleDownloadLegend`,
four download handlers + the cart handler as OrderScreen props, per-task state replacing
`packetDownloaded`); retargeted OrderScreen + App terminal-state tests.

Planner note (UI-SPEC deviation — done-state trigger, honest per D-07): the approved 26-UI-SPEC
"Interaction States" row reads "section collapses to 'Downloaded ✓' panel once all canvas files
fire." We deliberately DO NOT gate the section-① done-state on all four artifacts. Requiring all
four (grid PNG + grid+legend PNG + legend PNG + JSON packet) is too rigid and would UNDER-claim:
a customer who legitimately only needs the grid PNG + packet (and skips the standalone legend or
combined sheet) would never see "Downloaded ✓" even though everything they took is really on disk.
The old packet-only trigger was equally wrong in the OTHER direction — it OVER-claimed (packet
marks done though the PNGs were not taken) and UNDER-claimed (three PNGs but no packet showed
nothing). Chosen rule: `canvasDownloaded` becomes true on ANY section-① canvas download (any of the
three PNGs OR the JSON packet). This is the most faithful reading of D-07's stated intent — "the
files are really on the user's disk" — and of the section-① sub-copy ("Your canvas files are on
your device"), which is honestly true after any single download. The done-panel stays visible while
the download buttons remain available, so a customer can keep grabbing the other artifacts. This is
an explicit, honest deviation from the UI-SPEC's literal "all files fire" wording, recorded here so
it is visible rather than silently reinterpreted; the independence principle (canvas vs. cart) is
fully preserved.
</objective>
