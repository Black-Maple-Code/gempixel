<objective>
Run the one-time manual, in-browser verification that the relocation (a) keeps the UI responsive on a large image and (b) produces bit-identical matched output vs the pre-phase main-thread pipeline (D-11), plus (c) hard-fails cleanly on an unsupported browser (D-07).

Purpose: The resample byte-parity between `OffscreenCanvas` (worker) and `HTMLCanvasElement` (old main-thread path) is implementation-defined, not spec-guaranteed (RESEARCH A1–A4) — node/Vitest cannot exercise the decode path, so this manual gate is the only place parity is confirmed for the target browser. This is a one-time gate, NOT a permanent automated harness (the automated harness was explicitly deferred).
Output: A recorded pass/fail for responsiveness, parity, and the unsupported-browser banner, captured in the plan SUMMARY.
</objective>

  <name>Task 1: Manually verify responsiveness (D-09/PERF-01), bit-identical parity (D-11), and unsupported-browser hard-fail (D-07)</name>
  <what-built>Decode + resample (`drawImage` to the 2000px cap) + `getImageData` readback + `boxSampleImage` now run inside `matcher.worker.ts` on a transferred `ImageBitmap`; the hook creates the bitmap on the main thread; the loading overlay is phase-labeled ("Preparing image…" → "Matching colors: {n}%"); worker/decode failures and unsupported browsers surface through the stage-agnostic "Couldn't process the image:" banner.</what-built>
  <action>
    Run `npm run dev` and open http://localhost:5173.

    1. RESPONSIVENESS (PERF-01 / D-09): Load a LARGE source image (~4000×3000 or larger). While the match runs, confirm the UI stays interactive (the page does not freeze; you can move the pointer / scroll). Confirm the overlay shows an indeterminate "Preparing image…" first, then flips to the determinate "Matching colors: {n}%". Confirm the spinner and the error banner never appear at the same time.

    2. PARITY (D-11) — the bit-identical gate. Pick ONE fixture image and fixed settings (record exact cols, rows, base kit, substitution on/off + threshold, smoothing on/off + strength). With the CURRENT (new) build, load the fixture and capture the output as a diffable surface: the per-DMC supply counts in the report (and/or an exported grid PNG via the Option C download / print legend). Then build the PRE-PHASE baseline: from a clean checkout, `git worktree add ../gempixel-baseline <commit-before-phase-13>` (or stash + checkout that commit), `npm install` if needed, `npm run dev`, and load the SAME fixture with the SAME recorded settings. Capture the same surface. DIFF them: the per-DMC counts must match exactly and the grid must look pixel-for-pixel identical. If a handful of cells differ only near high-contrast edges, note it (RESEARCH Pitfall 1/2 — flag semi-transparent-PNG and EXIF-oriented cases specifically; test one opaque JPG and, if available, one EXIF-rotated photo and one semi-transparent PNG). Record the exact fixture + settings + result in the SUMMARY.

    3. UNSUPPORTED HARD-FAIL (D-07): Force the capability probe false — either temporarily call `__setOffscreenSupportForTest(false)` from a dev entry point, or in DevTools stub `OffscreenCanvas.prototype.getContext` to return null, then trigger a match. Confirm the banner reads "Couldn't process the image: … update your browser …", the spinner does NOT get stuck, and nothing crashes. Undo the override afterward.

    Record all three results (pass/fail + fixture/settings/notes) in `13-03-SUMMARY.md`. If parity fails, STOP and report — do not mark the phase verified; the fix is a code change back in Plan 13-01 (likely a `createImageBitmap` option or a stray `imageSmoothing` override), not a scope reduction.
  </action>
