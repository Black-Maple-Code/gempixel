<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| localStorage → checkout | The unmapped-colors log is read during checkout; a corrupt value (another tab / manual edit) makes `JSON.parse` throw and silently kills the flow. |
| error string → banner render | Failure messages (and any interpolated stored/error text) are rendered in the DOM; markup injection is possible if not text-encoded. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-11-06 | Tampering | `handleShopifyCheckout` unguarded `JSON.parse` (App.tsx:1011) | high | mitigate | Wrap the read in try/catch with `[]` fallback via `safeStorage`; corrupt value no longer aborts checkout, and the failure surfaces in the banner (W4, ASVS V5 input validation). |
| T-11-07 | Information Disclosure / Injection | `actionError` banner render | medium | mitigate | Banner renders `{actionError}` as a plain JSX text child — never `dangerouslySetInnerHTML` — so a crafted error/stored string cannot inject markup (ASVS output encoding). |
| T-11-08 | Denial of Service | download / checkout / save silent failure | medium | mitigate | Each failure path now sets `actionError` instead of a silent `console.error`-only no-op, so the user is informed rather than left with a dead control (ERR-01, W5). |
| T-11-SC | Tampering | npm/pip/cargo installs | high | accept | No package installs in this phase — pure code change; supply-chain surface unchanged. |
</threat_model>
